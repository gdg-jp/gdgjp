# M1 — P0 bugs + low-risk schema cleanup

> **Correction (2026-05-15):** M1's premise that the `verification` table is unused under `genericOAuth` is **wrong**. better-auth persists the OAuth `state` + PKCE `codeVerifier` to `verification` between the authorize redirect and the callback (see `dist/oauth2/state.mjs::generateState → setOAuthState`). The drop in migrations `tinyurl/0009`, `img/0003`, `scheduler/0004` was a regression; sign-in fails on prod with `D1_ERROR: no such table: verification`. Fix-forward migrations `tinyurl/0012`, `img/0006`, `scheduler/0006` recreate the table. The `account.password` drop in those same migrations stands — that column really is unused at OAuth-only RPs.


## Goal

Land in one PR:

1. **Audit #1:** Scheduler is missing from the IdP federated sign-out list.
2. **Audit #2:** RP sign-out has no `id_token_hint`, so a sign-out link from any source signs out whoever is logged in at the IdP (CSRF).
3. **Audit #9:** Trusted-client seed migrations duplicate the runtime `trustedClients` config and ship empty `clientSecret = ''` rows.
4. **Audit #10 (slice):** Drop `verification` table and the `password` column from `account` at each RP.

## Library finding that shapes M1

Verified in `node_modules/.pnpm/better-auth@1.6.9.../better-auth/dist/plugins/oidc-provider/index.mjs:1009-1086` and `dist/plugins/generic-oauth/routes.mjs:237`:

- `end_session_endpoint` verifies `id_token_hint` either via the JWT plugin (asymmetric) or — when `client_id` is also present — via HMAC with `client.clientSecret`. We don't load the JWT plugin, so HMAC is the verification path.
- `genericOAuth` always populates `account.idToken` at callback time. We can read it back at RP sign-out time to pass as the hint.
- `post_logout_redirect_uri` must match one of the client's registered `redirectUrls` exactly. Our env-configured `*_REDIRECT_URLS` today contain **only** the OAuth callback URL. We need to add a second URL per RP (the post-logout landing) before we can rely on `post_logout_redirect_uri`.

### Why we don't redirect straight to `end_session_endpoint` in M1

Today's `accounts/auth/signout` route is the *federated* sign-out — it renders hidden iframes pointing at each RP to clear their cookies. The standard `end_session_endpoint` only touches the IdP session + that one RP's `post_logout_redirect_uri`. Redirecting RPs at `end_session_endpoint` would lose the cross-RP iframe cleanup and make the situation worse than today.

So M1's fix for #2 is layered:

- RP continues to redirect to the IdP's federated `/auth/signout` route (preserving iframe cleanup of sibling RPs).
- RP attaches `id_token_hint` + `client_id` to that redirect.
- The IdP federated handler verifies the hint (HMAC with the client secret it already knows from `trustedClientsFromEnv`) and confirms the validated `sub` matches the current IdP session's user id. If not, refuse.

M3 will replace the iframe cleanup with a proper top-level chain through each RP's `frontchannel_logout_uri`, at which point the RP can redirect to `end_session_endpoint` directly.

## Changes

### `accounts/app/routes/auth.signout.ts`

- Add a `SCHEDULER_REDIRECT_URLS` line to the `rpOrigins(env)` aggregator (audit #1).
- Before delegating to `handleFederatedSignOut`, parse `id_token_hint` and `client_id` from the query string. If present:
  - Look up the client secret from `trustedClientsFromEnv(env)` keyed by `client_id`.
  - Verify the JWT with HMAC-SHA256 using that secret (use the `jose` library — already in `node_modules`).
  - Confirm the decoded `sub` equals the current IdP session user id (`auth.api.getSession`). If not, 400.
- If `id_token_hint` is absent, log a deprecation warning but still proceed (so existing RP code keeps working during rollout).

## Risks

- **`ALTER TABLE DROP COLUMN` on prod D1**: D1 supports it, but verify against the deployed db before merge by running `migrate:remote` in a staging environment.
- **`id_token_hint` HMAC verify** relies on the IdP's `oauthApplication.clientSecret`. Since clients are configured via `trustedClientsFromEnv` (env vars), not the seeded DB rows, the secret used for HMAC must match the secret better-auth used to sign the id_token at issuance. Confirm during testing that the HMAC verifies — if it doesn't, fall back to grabbing the secret the same way better-auth does internally.
- **Existing user sessions** without an `idToken` row (e.g., session predates idToken being stored): the redirect omits `id_token_hint` and the IdP logs the warning; behavior matches today. New logins start populating it.

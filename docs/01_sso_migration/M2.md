# M2 — Claim modeling

## Goal

- **Audit #4:** stop using the OIDC `profile` scope as a transport for app-specific claims (`chapter`, `chapters`, `isAdmin`). Move them onto a custom scope `gdgjp:chapters`.
- **Audit #5:** stop denormalizing `isAdmin` onto every RP's `user` table. Resolve it via fresh claims so revocation propagates without re-login.
- **Audit #6 additive piece — deferred.** Adding `user.idp_sub` was originally listed here; better-auth 1.6.9 exposes no post-OAuth-callback hook, so backfill-only would leave new users with NULL. Postponing until there's a concrete consumer (e.g. the FK rekey work).

## Investigation findings that shaped M2

Verified in `node_modules/.pnpm/better-auth@1.6.9.../better-auth/dist/`:

- **`oidcProvider({ scopes })`** option merges with the default `["openid","profile","email","offline_access"]` list (`plugins/oidc-provider/index.mjs:167-173`). The merged scope name passes through to `getAdditionalUserInfoClaim`'s second arg.
- **Discovery `scopes_supported` is hardcoded.** `.well-known/openid-configuration` won't list `gdgjp:chapters`. We're accepting this minor non-conformance; clients can still request the scope and it gates correctly.
- **`getAdditionalUserInfoClaim` receives the full user record** (including `additionalFields` like `isAdmin`), so the IdP can emit `isAdmin` without an extra DB read.
- **No post-OAuth-callback hook.** Confirmed via `dist/oauth2/link-account.mjs` — the path goes straight through `handleOAuthUserInfo` → `createOAuthUser`/`linkAccount`/`updateUser` with no extension point.
- **`additionalFields` removal is non-breaking.** `dist/db/schema.mjs::parseUserOutput` runs `filterOutputFields`, so an orphaned DB column is silently dropped from the user object. We can remove `isAdmin` from `buildRpAuth`'s `additionalFields` before (or alongside) dropping the DB column.
- **Today's `isAdmin` flow was dead end-to-end.** The IdP `getAdditionalUserInfoClaim` didn't include `isAdmin` in its returned object, so userinfo never carried it. RP `mapProfileToUser` mapped `profile.isAdmin === true` → always `false`. `isSuperAdmin(authUser)` at tinyurl/img consequently always returned false. M2 fixes this rather than removing the call sites.

## Changes

### IdP — `accounts/app/lib/auth.server.ts`

```ts
initializeIdpAuth({
  …,
  scopes: ["gdgjp:chapters"],
  getAdditionalUserInfoClaim: async (user, scopes) => {
    // Transitional: gate on either the new app scope or the legacy `profile`
    // scope so RPs that haven't yet been updated keep working.
    if (!scopes.includes("gdgjp:chapters") && !scopes.includes("profile")) return {};
    const chapterClaim = await getChapterClaim(env.DB, user.id);
    return { ...chapterClaim, isAdmin: user.isAdmin === true };
  },
});
```

The `profile` fallback is a transitional safety net for staggered deploys. A follow-up PR drops the fallback once every RP is confirmed to be on the new scope.

### Library — `gdg-lib/src/auth/server.ts`

- `IdpAuthConfig` gets `scopes?: string[]` and the `getAdditionalUserInfoClaim` user param widens to `{ id: string; isAdmin?: boolean | null }`. Both passed through to `oidcProvider`.
- `buildRpAuth`:
  - `genericOAuth.config[0].scopes` adds `"gdgjp:chapters"`.
  - `mapProfileToUser` no longer writes `isAdmin`.
  - `user.additionalFields.isAdmin` declaration removed.
- `buildIdpAuth` is unchanged — IdP user table still owns the real `isAdmin` column.

### RPs — fresh claims drive `user.isAdmin`

- `tinyurl/app/lib/chapter.server.ts::UserChapters` gains `isAdmin: boolean`. The 30s in-isolate cache value carries it.
- `tinyurl/app/lib/auth-redirect.ts::requireUserWithChapter` returns `{ user: { ...user, isAdmin: resolved.isAdmin }, … }`. Existing `isSuperAdmin(ctx.user)` call sites in `tinyurl/app/lib/permissions.ts` now see the live value.
- `img/app/lib/chapter.server.ts`: new `fetchChapterClaimsForUser` returns `{ chapter, isAdmin }`; old `fetchChapterForUser` becomes a thin wrapper (still used by `routes/no-chapter.tsx`).
- `img/app/lib/auth-redirect.ts::requireUserWithChapter` overrides `user.isAdmin` from `claims.isAdmin`. `isSuperAdmin(user)` call sites in `img/app/lib/permissions.ts` now see the live value.
- Scheduler has no `isSuperAdmin` call sites — no wiring change, just the migration drops the column.

### RP migrations

- `tinyurl/migrations/0010_drop_user_isadmin.sql` — `ALTER TABLE "user" DROP COLUMN "isAdmin"`
- `img/migrations/0004_drop_user_isadmin.sql` — same
- `scheduler/migrations/0005_drop_user_isadmin.sql` — same

Each app's `schema.sql` regenerated via `pnpm --filter @gdgjp/<app> schema:dump`.

## Verification

1. `pnpm typecheck` / `pnpm lint` / `pnpm test` clean.
2. `pnpm --filter @gdgjp/<app> migrate:local` succeeds for tinyurl, img, scheduler. `schema:dump` regenerated; `user.isAdmin` no longer present.
3. Local end-to-end:
   - Sign in fresh at tinyurl/img — confirm chapter info still resolves (via the new scope, transitionally also via `profile`).
   - Inspect the `id_token` payload at debug time: confirm it contains `chapterId`/`chapters`/`isAdmin` (gated on `gdgjp:chapters`).
   - Toggle `accounts.user.isAdmin` from `0` → `1` for a test user; hit a tinyurl link the user doesn't own; confirm `isSuperAdmin` bypass now applies (without re-login, after the 30s cache TTL expires).
4. Inspect IdP `.well-known/openid-configuration` — note that `scopes_supported` does NOT list `gdgjp:chapters` (documented limitation, not a regression).

## Risks

- **Staggered deploys.** If the IdP deploys before RPs are updated to request `gdgjp:chapters`, the `profile` fallback keeps everything working. If RPs deploy before the IdP, RPs request `gdgjp:chapters` which the IdP ignores — they still get chapter info via `profile` (since the legacy gate is still there). Either order is safe.
- **30s cache.** `isAdmin` revocation lags by up to 30 seconds at each Worker isolate. M4 will tighten this with a D1-backed session-level cache and invalidation hooks.
- **`accounts.user.isAdmin` is still set via a manual SQL operation** (no admin UI today). Out of scope here; flagged for later.

## Follow-ups

- Remove the `profile`-fallback gate in `accounts/app/lib/auth.server.ts` after all RPs are on the new scope.
- Defer audit #6 (`user.idp_sub`) until the FK rekey work has a real consumer.

# M3 — RP-Initiated Logout (top-level redirect chain)

## Goal

- **Audit #3:** decommission the iframe-based federated logout. Iframes pointing at each RP's `/auth/signout-iframe` no longer reliably send the RP's `SameSite=Lax` session cookies (Safari ITP, Chrome's third-party-cookie phase-out). Replace with a top-level redirect chain.

## Approach

The OIDC Front-Channel Logout spec uses iframes — exactly the pattern we're moving away from. The spec-conformant alternative (Back-Channel Logout, server-to-server POSTs) isn't supported by better-auth out of the box. So M3 implements a **non-standard but third-party-cookie-immune redirect chain**: the IdP navigates the user's main browser frame through each RP's logout URL serially.

### Flow

```
[user clicks Sign out at tinyurl]
   │
   ▼
RP /auth/signout                            (unchanged from M1)
   │  302 with id_token_hint + client_id
   ▼
IdP /auth/signout                           (M1: validate id_token_hint)
   │  ┌── mints chain JWT: {step:0, rps:[rp1,rp2,…], returnTo, frontchannelPath}
   │  └── 302 to rp1/auth/frontchannel-logout?iss=<idp>&continue=<idp>/auth/signout/advance?state=<jwt>
   ▼
RP1 /auth/frontchannel-logout               (new)
   │  ┌── validates iss origin == IdP origin
   │  ├── validates continue origin == IdP origin
   │  ├── auth.api.signOut → Set-Cookie clearing
   │  └── 302 to continue URL
   ▼
IdP /auth/signout/advance                   (new)
   │  ┌── verifies state JWT (HS256 + BETTER_AUTH_SECRET, exp 5min)
   │  ├── increments step
   │  └── 302 to rp2/auth/frontchannel-logout?iss=<idp>&continue=<idp>/auth/signout/advance?state=<next-jwt>
   ▼
…  (repeat until step >= rps.length)
   ▼
IdP /auth/signout/advance (final)
   │  ┌── calls auth.api.signOut at IdP → clears IdP cookie
   │  └── 302 to validated returnTo
```

### State JWT

- Algorithm: HS256, signed with `config.secret` (= `BETTER_AUTH_SECRET`).
- Claims: `{step, rps, returnTo, frontchannelPath, iat, exp}`. `exp = iat + 5 min`.
- Only the IdP signs and verifies. RPs treat it as opaque and forward it via the `continue` query param after their cookie clear.

### Security model

- RP `handleFrontchannelLogout`: rejects requests whose `iss` origin doesn't match the configured IdP origin (`config.idp.url`), AND whose `continue` URL origin doesn't match the same IdP origin. Both checks together prevent (a) hostile sites triggering RP logouts and (b) the RP being weaponized as an open redirect.
- IdP `handleAdvanceSignOut`: state JWT verification (HMAC + exp). Invalid/missing state → redirect to fallback (`/signin`) — best-effort, not 500.
- IdP `handleFederatedSignOut`: `return_to` is still validated against `appUrl ∪ rpOrigins` (existing `safeReturnTo`).

### What's deleted

- `gdg-lib/src/auth/server.ts`: `handleSignOutIframe`, `renderFederatedSignOutPage`, `frameAncestorsCsp` — all iframe-specific.
- Per-RP routes: `tinyurl/app/routes/auth.signout-iframe.ts`, `img/.../auth.signout-iframe.ts`, `scheduler/.../auth.signout-iframe.ts`.
- The corresponding `auth/signout-iframe` paths in each RP's `routes.ts`.
- `AuthInstance.handleSignOutIframe` interface member.

### What's added

- `gdg-lib`: `handleAdvanceSignOut` (IdP chain step), `handleFrontchannelLogout` (RP cookie clear). Internal `signLogoutState`/`verifyLogoutState` (HS256 JWT helpers).
- `accounts/app/routes/auth.signout.advance.ts` registered as `/auth/signout/advance`.
- Per-RP routes: `auth.frontchannel-logout.ts` at path `/auth/frontchannel-logout`.

### Convention over config

RP frontchannel URLs are derived as `<rp-origin>/auth/frontchannel-logout`. No new wrangler env vars needed; the IdP's existing `*_REDIRECT_URLS`-derived origins are reused.

## Verification

1. `pnpm typecheck` / `pnpm lint` / `pnpm test` clean.
2. Manual trace via `curl -I` against `pnpm dev`:
   - `curl -I "http://localhost:PORT/auth/signout?id_token_hint=…&client_id=…"` at the IdP → 302 Location is `<rp1>/auth/frontchannel-logout?iss=…&continue=<idp>/auth/signout/advance?state=…`
   - `curl -I "<rp1>/auth/frontchannel-logout?iss=<idp>&continue=<idp>/…"` → 302 to continue URL, with `Set-Cookie` clearing the session.
   - Tamper with the JWT (flip one char) → advance route 302s to fallback `/signin`, not 500.
   - Tamper with `iss` (use a different origin) → RP returns 400.
   - Tamper with `continue` (point to evil.com) → RP returns 400.
3. End-to-end browser trace: sign in at tinyurl, img, scheduler in three tabs (same browser session). Hit IdP `/auth/signout?return_to=…`. Confirm the address bar walks through each RP's frontchannel URL and lands at `return_to`, and all four cookies are cleared (DevTools → Application → Cookies).
4. Block third-party cookies in the browser settings; repeat (3). Should still work — that's the whole point.

## Risks & follow-ups

- **Chain interruption.** If the user closes the tab mid-chain, downstream RPs aren't logged out. Their cookies expire on the 5-min `cookieCache.maxAge` (M4 may tighten this with D1-backed session-claims caching).
- **Discovery non-conformance.** `frontchannel_logout_supported: true` won't appear in `.well-known/openid-configuration` (better-auth's discovery is hardcoded — same posture as M2's `scopes_supported`). Documented and accepted.
- **Open redirect surface.** RPs validate `continue` URL origin == IdP origin. The IdP is the only party that crafts continue URLs. If the IdP origin is ever compromised, an attacker could chain RP redirects — but at that point the entire SSO is moot anyway.
- **JWT replay.** A leaked state JWT lets an attacker re-trigger advance steps within 5 minutes. The worst outcome is an extra IdP logout (idempotent) and an unexpected redirect to whatever `returnTo` the original chain encoded. Low impact.

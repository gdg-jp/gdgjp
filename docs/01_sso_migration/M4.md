# M4 — Userinfo cache + secret hashing

> **Correction (2026-05-15):** The `storeClientSecret: "hashed"` part of M4 had to be reverted. better-auth's token endpoint runs `hash(client_secret_from_request) === stored_client_secret`. Our `trustedClients` come in from env as plaintext and better-auth stores them as-is, so the compare never matches → `invalid_client` on every callback. Worse, `client.clientSecret` is *also* used directly as the HS256 HMAC key for ID-token signing and `id_token_hint` verification (`oidc-provider/index.mjs:622, 1023`), so pre-hashing the trustedClient secrets isn't an option either — it would break those signing paths. With static trustedClients (no dynamic `/oauth2/register`), `"plain"` is the only viable mode. Reverted in PR #53. The userinfo-cache part of M4 stands.


## Goal

- **Audit #7:** stop hitting `/oauth2/userinfo` once per cold isolate. The existing in-memory `Map` caches at `tinyurl/app/lib/chapter.server.ts` and `img/app/lib/chapter.server.ts` evaporate when a Worker isolate cycles, so every cold request to a chapter-gated route triggers a userinfo round-trip and possibly a refresh_token grant.
- **Audit #8:** flip the IdP's `storeClientSecret` from `"plain"` to `"hashed"`. Defense-in-depth — `oauthApplication` is currently empty (the M1 seed migrations removed the placeholder rows; trustedClients come in from env), so it's a one-line change with no data migration.

## Approach — userinfo cache

A new per-RP `userinfo_cache` table:

```sql
CREATE TABLE userinfo_cache (
  user_id     TEXT NOT NULL PRIMARY KEY REFERENCES "user"(id) ON DELETE CASCADE,
  claims_json TEXT NOT NULL,
  fetched_at  INTEGER NOT NULL
);
```

Migrations: `tinyurl/migrations/0011_userinfo_cache.sql`, `img/migrations/0005_userinfo_cache.sql`. Scheduler doesn't call `getFreshClaims` (no chapter-gated routes today), so no migration there.

`gdg-lib/src/auth/server.ts::fetchUserClaims` is restructured:

1. If a same-isolate inflight promise exists, return it (existing dedup, unchanged).
2. Read `userinfo_cache` row for the user. If present and `Date.now() - fetched_at < 5 min`, parse JSON and return — no IdP hit.
3. Otherwise: fetch account tokens, refresh access token if needed, call userinfo. UPSERT the response JSON into `userinfo_cache` with the current timestamp.

Cache-write failures don't fail the request — logged and swallowed.

The 30s in-isolate `Map` caches at `tinyurl/app/lib/chapter.server.ts` and `img/app/lib/chapter.server.ts` are removed (the D1 cache supersedes them). The library-level inflight dedup `Map` stays — it prevents two concurrent requests on the same isolate from both hitting D1 and the IdP.

### Why a separate table

- Tying claims to the `session` row would couple a user-level value to a session-level lifecycle and break multi-session users out of cache.
- Adding columns to the `user` row brings back the denormalization we just removed in M2.
- A scratch `userinfo_cache` table makes "this is a cache" explicit, supports `ON DELETE CASCADE` for auto-invalidation, and can be truncated wholesale if needed.

### Invalidation

- User logout (M3 chain): RP `auth.api.signOut` doesn't touch `userinfo_cache`. The next hit after re-login is fresh, so no explicit invalidation is required — but stale data won't outlive 5 min anyway.
- User deletion: `ON DELETE CASCADE` clears the cache row.
- Admin-toggle / chapter change: at worst 5-min staleness. M3-level webhook-based invalidation is out of scope here.

## Approach — secret hashing

`gdg-lib/src/auth/server.ts::buildIdpAuth` switches `storeClientSecret: "plain"` → `"hashed"`. better-auth uses SHA-256 + base64url (verified in `node_modules/.../dist/plugins/oidc-provider/utils.mjs::defaultClientSecretHasher`).

No data migration — `oauthApplication` was verified empty via `wrangler d1 execute --command "SELECT ... FROM oauthApplication"`. The trustedClients passed in from env via `trustedClientsFromEnv` are kept in memory by better-auth at construction time and aren't subject to `storeClientSecret`.

If `/oauth2/register` is ever wired up later, new rows will be hashed.

1. `pnpm typecheck` / `pnpm lint` / `pnpm test` clean.
2. `pnpm --filter @gdgjp/{tinyurl,img} migrate:local` applies cleanly; `schema:dump` shows `userinfo_cache` table at both apps.
3. Manual: in `pnpm dev`, sign in at tinyurl, hit a chapter-gated route. Inspect logs — first request shows a userinfo fetch; subsequent requests within 5 min show no userinfo hit. Inspect `userinfo_cache` via `wrangler d1 execute --local --command "SELECT user_id, length(claims_json), fetched_at FROM userinfo_cache"` — row exists with current timestamp.
4. Cold-isolate test: kill the dev server, restart, hit the same chapter-gated route. Confirm no userinfo fetch (cache hit from D1).
5. After 5 min, repeat — confirms refresh round-trip.
6. Negative path: stop the IdP, sign-in still works at the RP (cache hit), but past-TTL requests fail-open with `ClaimsUnavailableError` and redirect to `/signin` (unchanged from today).

## Risks & follow-ups

- **Stale `isAdmin` for up to 5 min** after admin-toggle. Acceptable for our scale; if it ever isn't, a webhook from accounts → RP `DELETE FROM userinfo_cache WHERE user_id = ?` would tighten it.
- **D1 write per cache miss.** Cheap, but at very high QPS the inflight-dedup map already coalesces.
- **No automatic eviction.** Rows live until the user is deleted. Worst case the table grows by one row per ever-signed-in user — fine for our scale; a future cron-style cleanup is possible if needed.

# M4 — Userinfo cache + secret hashing (deferred)

## Goal

- **Audit #7:** stop hitting IdP `/oauth2/userinfo` per request. Current 30-second in-memory cache (`tinyurl/app/lib/chapter.server.ts`) doesn't survive Worker isolate boundaries.
- **Audit #8:** switch IdP to `storeClientSecret: "hashed"`. Today secrets sit in plain text in `oauthApplication.clientSecret`.

## Approach — userinfo caching

Move the cache from per-isolate Maps to the `session` table:

- Add `claims_json TEXT NULL` and `claims_fetched_at INTEGER NULL` to RP `session` schema.
- `gdg-lib/src/auth/server.ts::fetchUserClaims` checks `session.claims_fetched_at` first; if within TTL (5 min), parse and return `claims_json`. Otherwise refresh from IdP and persist.
- Invalidation: M3's RP-Initiated Logout already clears the session row. For chapter changes, expose an IdP webhook → RP that nulls `claims_fetched_at`.

## Approach — secret hashing

- `accounts/app/lib/auth.server.ts` → flip `storeClientSecret: "plain"` to `"hashed"`.
- One-shot rehash script: `accounts/scripts/rehash-client-secrets.ts` reads each `oauthApplication` row, SHA-256s the `clientSecret`, writes it back. Document as a manual step (not a D1 migration) so we can rerun safely.
- No-op for runtime clients defined via `trustedClientsFromEnv` since those are passed in-memory; the IdP hashes them on first use.

## Files

Library:
- `gdg-lib/src/auth/server.ts::fetchUserClaims` — swap in-memory map for D1-backed cache.

Per-RP migrations:
- Add `claims_json`, `claims_fetched_at` columns to `session`.

IdP:
- `accounts/app/lib/auth.server.ts` — `storeClientSecret: "hashed"`.
- `accounts/scripts/rehash-client-secrets.ts` — one-shot.

## Verification (when M4 ships)

- Hit the same chapter-gated route twice on cold isolates → second request reads claims from D1, no userinfo round-trip.
- After rehash script run, inspect `oauthApplication.clientSecret` rows are SHA-256 base64url; IdP-initiated OAuth flows still succeed.

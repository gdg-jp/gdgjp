## medium: Comment drafts are cleared after unrelated or failed comment actions

id: fnd_sig-feat-library-053e550e39-f76b_74c9ab9e90
category: data-loss
confidence: high
triage: confirmed-bug
status: open
feature: Node source wiki/app/components#1 (feat_library_053e550e39)

evidence:
- wiki/app/components/CommentEditor.tsx:65-73 (CommentEditor)
- wiki/app/components/CommentSection.tsx:24-25 (CommentSection)
- wiki/app/components/CommentSection.tsx:47-58 (handleDelete/handleToggleReaction)

CommentEditor resets its local draft whenever the passed isSubmitting prop transitions from true to false, without knowing whether this editor submitted successfully. CommentSection passes one shared fetcher state to the top-level editor and reply editors, and the same fetcher is also used for delete and reaction actions. If a user has a draft open and toggles a reaction or deletes another comment, the fetcher goes busy then idle and the draft editor clears. The same reset also happens after a failed add-comment request because no success signal is checked.

recommendation:
Reset a CommentEditor only after that specific editor's submit succeeds. Separate fetchers per action/editor or pass an explicit successfulSubmissionKey/result instead of a shared busy boolean. Do not clear content on failed submissions or unrelated comment actions.

test analysis:
The included tests only cover TipTapRenderer rendering and TOC extraction; they do not mount CommentSection/CommentEditor or exercise fetcher state transitions.

suggested regression test:
Add a CommentSection component test with a mocked fetcher: enter draft text, trigger handleToggleReaction or handleDelete, simulate fetcher returning to idle, and assert the editor content remains unchanged. Add a failed addComment transition case that also preserves the draft.

minimum fix scope:
CommentEditor reset contract and CommentSection fetcher/state wiring for add comment, add reply, delete, and reaction actions.

repro:
Type a new top-level comment without submitting it, click an existing comment reaction, wait for the request to finish, and observe the draft editor reset to empty.

## medium: Dirty-page navigation is blocked without any recovery UI

id: fnd_sig-feat-library-0dd0c5aab3-338c_387f8dea10
category: bug
confidence: high
triage: confirmed-bug
status: open
feature: Node package @gdgjp/wiki (feat_library_0dd0c5aab3)

evidence:
- wiki/app/components/PageEditor.tsx:146-147 (PageEditor)
- wiki/app/components/PageEditor.tsx:196-198 (PageEditor)

React Router's useBlocker returns a blocker object that must be handled by the component. This component calls useBlocker but discards the returned blocker, so when the editor is dirty and the user clicks the back link or another client-side route, navigation is blocked silently with no prompt, proceed, or reset path. That traps the user on the edit screen until they save/autosave or manually clear the dirty state.

recommendation:
Store the blocker returned by useBlocker and render a discard-changes confirmation when blocker.state is "blocked". On confirm call blocker.proceed(); on cancel call blocker.reset().

test analysis:
The included test list focuses on renderer and library/server helpers; no supplied test exercises PageEditor dirty navigation or React Router blocker behavior.

suggested regression test:
Add a component-level test for PageEditor with a memory router: mutate content/title, click the back link, assert a discard dialog appears, then assert cancel keeps the route and confirm proceeds.

minimum fix scope:
wiki/app/components/PageEditor.tsx

repro:
Open an existing page in PageEditor, edit the title or content so isDirty becomes true, then click the back link. The route transition is blocked, but no confirmation dialog appears and the user cannot choose to discard changes from this navigation attempt.

repro:

## medium: Archived popover cache can expose another user's archived page list

id: fnd_sig-feat-library-053e550e39-464d_071c1e5060
category: security
confidence: medium
triage: risk
status: open
feature: Node source wiki/app/components#1 (feat_library_053e550e39)

evidence:
- wiki/app/components/ArchivedContent.tsx:31-32 (cachedArchived)
- wiki/app/components/ArchivedContent.tsx:40-56 (ArchivedContent)
- wiki/app/components/ArchivedContent.tsx:58-61 (ArchivedContent)
- wiki/app/components/ArchivedContent.tsx:90-102 (ArchivedContent)

The archived data cache is module-scoped and not keyed by current user, session, or permissions. On reopen, the component renders cached pages and cached authorization flags immediately while the fresh /api/archived request is still loading. In a long-lived browser session where the authenticated user changes, a later user can temporarily see the previous user's archived page titles and admin/lead affordances before the new fetch completes. Server-side action authorization may block mutations, but the page metadata has already been disclosed in the UI.

recommendation:
Keep the cache component-local or key it by authenticated user and permission context. Alternatively clear cachedArchived on auth changes and show the skeleton while a fresh request is pending instead of rendering stale cross-session data.

test analysis:
The linked tests are TipTapRenderer-only and do not cover ArchivedContent caching, user changes, or stale authorization data.

suggested regression test:
Add a component test that seeds one user's archived response, rerenders/open as another user before the second request resolves, and asserts the first user's pages and permission buttons are not rendered.

minimum fix scope:
ArchivedContent cache ownership and loading behavior around /api/archived data.

repro:
As an admin/user A, open the archived popover so cachedArchived is populated. Change authentication to user B in the same SPA/runtime, then open the archived popover; the previous cache is used until /api/archived returns.

## medium: Null email claims collapse distinct users into the same local account

id: fnd_sig-feat-library-0bd5f1b7fc-212a_133f9afe43
category: security
confidence: medium
triage: risk
status: open
feature: Node package @gdgjp/gdg-lib (feat_library_0bd5f1b7fc)

evidence:
- gdg-lib/src/auth/rp.ts:320-321 (upsertUser)
- gdg-lib/src/auth/rp.ts:326-329 (upsertUser)
- gdg-lib/src/auth/rp.ts:427 (parseClaims)
- gdg-lib/README.md:134

The public UserClaims type and parseClaims allow email to be null, but upsertUser converts that to the empty string and uses it as the unique lookup key. If the IdP ever returns a valid /userinfo response without an email claim, every such identity maps to the same RP-local user row with email ''. That can merge authorization state and domain-owned data across unrelated identities. If email is mandatory for this auth system, the callback should reject missing/empty email before upsert instead of treating it as a stable identifier.

recommendation:
In handleCallback/upsertUser, require a non-empty email claim before creating or looking up the local user, and return an authentication failure when it is missing. Alternatively key users by a stable IdP subject plus provider if email-less users must be supported.

test analysis:
The included tests cover cookie helpers and the generic session mapper only; there are no rp.ts tests for parseClaims, callback handling, or upsert behavior with missing email claims.

suggested regression test:
Add an rp.ts unit test that feeds userinfo claims without email into the callback/upsert path and asserts authentication is rejected and no user row with email '' is created.

minimum fix scope:
gdg-lib/src/auth/rp.ts callback/upsert validation plus focused rp.ts tests.

## low: Malformed percent-encoding in Cookie headers can crash auth request handling

id: fnd_sig-feat-library-0bd5f1b7fc-363c_98a172ae4d
category: bug
confidence: high
triage: confirmed-bug
status: open
feature: Node package @gdgjp/gdg-lib (feat_library_0bd5f1b7fc)

evidence:
- gdg-lib/src/auth/cookie.ts:76 (parseCookies)
- gdg-lib/src/auth/cookie.test.ts:67-81 (parseCookies tests)
- gdg-lib/src/auth/rp.ts:255 (getSessionUser)

decodeURIComponent throws URIError for malformed percent escapes such as '%E0%A4%A'. parseCookies does not catch that exception, and rp.ts calls parseCookies while reading session cookies on normal auth paths. A client can send a malformed Cookie header and turn otherwise safe unauthenticated/session reads into an unhandled exception instead of a null session or ignored malformed cookie.

recommendation:
Catch decodeURIComponent failures per cookie entry and skip the malformed cookie value, or preserve the raw value when decoding fails.

test analysis:
The parseCookies tests cover missing equals signs and empty names, but not invalid percent-encoded values that make decodeURIComponent throw.

suggested regression test:
Add a parseCookies test with an invalid percent escape and assert it does not throw and ignores that cookie entry.

minimum fix scope:
gdg-lib/src/auth/cookie.ts parseCookies plus one cookie.test.ts regression.

repro:
Call parseCookies('session=%E0%A4%A') or request /api/auth/me with Cookie: <session-name>=%E0%A4%A; decodeURIComponent throws before the auth handler can return a normal JSON response.

category: bug

status: open

- wiki/app/components/BaseSidebar.tsx:60-70 (onDragHandleMouseDown)
- wiki/app/components/BaseSidebar.tsx:75-81 (cleanup effect)

During resize, the component mutates global body styles to force a resize cursor and disable text selection. Those styles are restored only in onMouseUp. Add a component test that fires mouseDown on the drag handle, unmounts the component, and asserts document.body.style.cursor and document.body.style.userSelect are restored.

minimum fix scope:
BaseSidebar drag lifecycle cleanup.

## low: Client accepts unvalidated persisted UI language

id: fnd_sig-feat-cli-command-15e7ddedfd-_5e0e5760f4

confidence: medium
triage: risk
status: open
feature: Node source wiki/app/:cli (feat_cli-command_15e7ddedfd)

- wiki/app/entry.client.tsx:46-49

The client applies the localStorage value directly as an i18next language even though the rest of the initialization is constrained to supported languages. Because localStorage is user-controlled and can contain stale or arbitrary values, a browser with ui_lang set to an unsupported value can switch the client i18n instance away from the SSR-supported locale after hydration, producing fallback or inconsistent language state in components that branch on i18n.language.

recommendation:
Validate savedLang against supportedLngs before calling changeLanguage, and optionally remove invalid stored values. Wrap localStorage access in a small try/catch if the app should tolerate storage-denied browsers.

The linked Vitest tests cover renderer, chunking, ingestion, Gemini schemas, Google Forms utilities, and page access helpers, but none exercise wiki/app/entry.client.tsx hydration or persisted language restoration.

suggested regression test:
Add a client-entry or extracted helper test that seeds ui_lang with an unsupported value and asserts changeLanguage is not called unless the value is included in supportedLngs.

minimum fix scope:
wiki/app/entry.client.tsx

repro:
In a browser console before loading the app, set localStorage.setItem("ui_lang", "fr") and reload. The client entry calls i18next.changeLanguage("fr") even though only the app-supported languages should be selectable.

## low: Share dialog uses dynamic Tailwind size classes that may not be generated

id: fnd_sig-feat-library-14b5af1b68-f5eb_101e9167ef
category: build-release
confidence: medium
triage: risk
status: open
feature: Node source wiki/app/components#2 (feat_library_14b5af1b68)

evidence:
- wiki/app/components/ShareDialog.tsx:52-61 (Avatar)

Tailwind class extraction depends on statically visible class names. Building class names like h-${size} and w-${size} can omit required avatar dimensions from the generated CSS, especially for values that are only present through this helper. That causes share-dialog avatars or pending-user chips to render with missing width/height in production builds.

recommendation:
Replace the dynamic class construction with an explicit size map, for example {6: 'h-6 w-6', 8: 'h-8 w-8'}, or use inline styles for dimensions.

test analysis:
The linked tests render TipTapRenderer only; they do not build Tailwind output or render ShareDialog avatar sizes.

suggested regression test:
Add a ShareDialog render test for pending-user and search-result avatars, or a build/CSS smoke test that verifies the generated stylesheet includes the avatar size classes used by this component.

minimum fix scope:
Change the local Avatar helper in ShareDialog.tsx to use statically enumerable size classes.
